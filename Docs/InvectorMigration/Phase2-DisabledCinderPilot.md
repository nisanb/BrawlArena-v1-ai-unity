## Serialized Reference Strategy

The builder starts from the generated Cinder prefab so its original skeleton paths and Avatar remain intact. It copies the selected root component configuration in two passes:

1. create every destination root component and build a source-root object/component map;
2. remap every serialized object reference, including UnityEvent targets, to the destination equivalent and clear references into template children that were deliberately not retained.

Native Rigidbody/Capsule values are copied explicitly rather than through `EditorUtility.CopySerialized`. The builder then applies the dormant safety values and rejects any remaining reference into the vendor template hierarchy. The focused test also rejects Unity missing-reference instance IDs.

```text

  -> no MonoBehaviour Awake/OnEnable/Start
  -> disabled vShooterMeleeInput
```

There is no `LegacyBrawlerAnimationDriver`, concrete Invector animation driver, `BrawlerController`, `PlayerBrawlerInput`, or Brawl `Health` baked into the prefab. The latter remain assembler-owned. `vThirdPersonController` inherits `vHealthController`, so it is serialized immortal and dormant; external Invector damage/death components are absent.
